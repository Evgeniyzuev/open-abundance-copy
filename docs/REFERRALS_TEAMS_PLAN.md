# Referrals And Teams Implementation Plan

План описывает внедрение реферальных ссылок, фиксации связи `реферер -> реферал` после регистрационного челленджа и командной структуры.

## Цели

- Пользователь может пригласить нового человека по персональной реферальной ссылке.
- Новый пользователь может прийти без реферальной ссылки; тогда после регистрации он распределяется через общую очередь.
- Новый пользователь может зайти по ссылке до регистрации, пройти guest-first сценарий и сохранить реферальный источник локально.
- Связь `реферер -> реферал` становится серверной и авторитетной только после завершения регистрационного челленджа.
- Реферал автоматически попадает в команду к пригласившему, если это разрешено правилами команды.
- Если у пригласившего нет места, система подбирает подходящего лида в дереве его команды ниже по структуре.
- Если внутри команды пригласившего нет подходящего места, пользователь попадает в общую очередь распределения.
- У каждого пользователя есть лид; исключение - пользователь с самым высоким уровнем, у него лидом считается система.
- Подписка на ленту лида создается автоматически и не может быть отменена, пока действует командная связь.

## Базовые правила

1. Члены команды всегда ниже уровнем, чем лид.
2. Лид всегда выше уровнем, чем каждый прямой участник его команды.
3. У каждого пользователя должен быть лид, кроме пользователя с самым высоким уровнем; его лид - система.
4. Вместимость команды считается не количеством мест, а суммой уровней прямых участников.
5. Базовая capacity лида равна `leader_capacity = leader.level * 10`.
6. Стоимость участника для capacity равна его уровню: `member_capacity_cost = member.level`.
7. Пользователи уровня `0` не распределяются в команды; командное распределение начинается с уровня `1`.
8. Реферал в первую очередь попадает в команду к тому, кто его пригласил.
9. Если у пригласившего не хватает свободной capacity или уровень реферала не ниже уровня пригласившего, система ищет место в дереве команды пригласившего ниже по структуре.
10. Если внутри команды пригласившего нет подходящего лида, реферал попадает в общую очередь распределения.
11. Пользователь без реферальной ссылки после регистрации сразу попадает в общую очередь распределения.
12. В общей очереди MVP-распределение может быть простым: первый подходящий лид со свободной capacity и уровнем выше пользователя.
13. Позже очередь можно улучшить приоритетами: меньшая загрузка лида, успешность команды, активность лида, retention участников, качество помощи новичкам.
14. Лида можно сменить не чаще одного раза в сутки.
15. Смена лида допустима только на лида выше уровнем.
16. Автоматическую подписку на посты и ленту текущего лида нельзя отменить вручную.
17. При смене лида старая обязательная подписка снимается или переводится в обычную, а новая обязательная подписка создается автоматически.
18. Одна пара `реферер -> реферал` фиксируется один раз и не переписывается обычным клиентским действием.
19. Лид получает в свой Core `10%` от суточного прироста Core прямых участников команды.
20. Командный бонус начисляется раз в 24 часа после daily rate, а не на каждую отдельную операцию роста Core.
21. Начисление идет снизу вверх: сначала участники нижних уровней, затем их лиды, чтобы бонусы, полученные снизу, вошли в суточный прирост лидов.

## Реферальная ссылка

Формат MVP:

```text
https://app.example.com/?ref=<referral_code>
```

Альтернативный читаемый формат:

```text
https://app.example.com/r/<referral_code>
```

Рекомендуемая модель:

- у каждого зарегистрированного пользователя есть стабильный `referral_code`;
- код не должен быть равен `user_id`;
- код можно показывать в UI и копировать в буфер;
- при компрометации или злоупотреблениях код можно перевыпустить, но старые коды лучше хранить как алиасы для аналитики и обратной совместимости.

## Хранение реферала до регистрации

Когда новый пользователь открывает приложение по реферальной ссылке:

1. Клиент читает `ref` из URL.
2. Клиент валидирует формат кода локально: длина, допустимые символы.
3. Клиент сохраняет pending referral в local-first хранилище гостя.
4. Если guest identity уже существует, referral привязывается к этому гостю.
5. Если pending referral уже есть, новая реферальная ссылка до регистрации может перезаписать предыдущую.
6. Для аналитики можно хранить историю referral touches, но авторитетным pending referral считается последний сохраненный код.
7. После сохранения referral параметр можно убрать из URL через `history.replaceState`, чтобы не таскать код по навигации.

Локальное поле в IndexedDB `guestIdentity`:

```ts
type PendingReferral = {
  referralCode: string;
  capturedAt: string;
  landingPath: string;
  utm?: {
    source?: string;
    medium?: string;
    campaign?: string;
  };
};
```

Важно: локальная запись не является авторитетной. Она только переносит намерение приглашения через guest-first путь до момента регистрации.

## Фиксация после регистрационного челленджа

Серверная связь создается не в момент первого открытия ссылки, а после завершения регистрационного челленджа `Save Your Progress`.

Поток с реферальной ссылкой:

1. Пользователь открывает приложение по `?ref=<code>`.
2. Приложение создает или загружает local guest identity.
3. Pending referral сохраняется в IndexedDB вместе с guest identity.
4. Пользователь проходит первые действия в guest mode.
5. Пользователь выполняет челлендж регистрации.
6. После успешного Supabase Auth callback приложение создает или загружает профиль, Core и Wallet.
7. Клиент отправляет на сервер claim-запрос: `pendingReferral`, `guestId`, `userId` из auth session.
8. Сервер находит владельца `referral_code`.
9. Сервер проверяет, что пользователь не приглашает сам себя.
10. Сервер проверяет, что у нового пользователя еще нет зафиксированного реферера.
11. Сервер создает запись `referral_edges`.
12. Сервер запускает распределение в команду.
13. Сервер создает обязательную подписку на лида.
14. Клиент помечает локальный pending referral как claimed.

Поток без реферальной ссылки:

1. Пользователь открывает приложение обычным способом.
2. Приложение создает или загружает local guest identity без `pendingReferral`.
3. Пользователь выполняет челлендж регистрации.
4. После успешного Supabase Auth callback приложение создает или загружает профиль, Core и Wallet.
5. Сервер не создает `referral_edges`.
6. Сервер запускает распределение пользователя через общую очередь.
7. После назначения лида сервер создает обязательную подписку на лида.

Claim должен быть идемпотентным: повторный запрос после перезагрузки или сетевого сбоя не должен создавать дубль.

## Предлагаемая серверная схема

Для MVP не создаем отдельную PostgreSQL schema `growth` и не создаем отдельную таблицу `teams`.

Причины:

- команда в текущей модели - это прямые участники конкретного лида, поэтому `leader_user_id` уже является естественным идентификатором команды;
- отдельный `team_id` добавляет таблицу и связи, но не дает пользы, пока у команды нет отдельного названия, настроек, брендинга, приватности или истории владения;
- меньше таблиц проще поддерживать в RLS, типах Supabase и UI;
- если позже появятся самостоятельные команды/клубы, таблицу `teams` можно добавить миграцией и связать существующие memberships с новым `team_id`.

Минимальный набор таблиц:

```sql
create table public.referral_codes (
  code text primary key,
  user_id uuid not null references auth.users(id) on delete cascade,
  is_active boolean not null default true,
  created_at timestamptz not null default now(),
  retired_at timestamptz,
  unique (user_id, code)
);

create table public.referral_edges (
  referral_user_id uuid primary key references auth.users(id) on delete cascade,
  referrer_user_id uuid not null references auth.users(id) on delete restrict,
  referral_code text references public.referral_codes(code),
  guest_id uuid,
  captured_at timestamptz,
  claimed_at timestamptz not null default now(),
  source text,
  check (referral_user_id <> referrer_user_id)
);

create table public.team_memberships (
  member_user_id uuid primary key references auth.users(id) on delete cascade,
  leader_type text not null default 'user' check (leader_type in ('user', 'system')),
  leader_user_id uuid references auth.users(id) on delete restrict,
  assigned_by text not null check (assigned_by in ('direct_referral', 'team_fallback', 'global_queue', 'manual_change')),
  assigned_at timestamptz not null default now(),
  last_leader_change_at timestamptz,
  is_active boolean not null default true,
  check (
    (leader_type = 'user' and leader_user_id is not null and member_user_id <> leader_user_id)
    or
    (leader_type = 'system' and leader_user_id is null)
  )
);

create table public.team_assignment_queue (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  preferred_referrer_user_id uuid references auth.users(id) on delete set null,
  status text not null default 'pending' check (status in ('pending', 'assigned', 'failed', 'cancelled')),
  reason text,
  created_at timestamptz not null default now(),
  processed_at timestamptz
);

create table public.team_membership_events (
  id uuid primary key default gen_random_uuid(),
  member_user_id uuid not null references auth.users(id) on delete cascade,
  old_leader_user_id uuid references auth.users(id) on delete set null,
  new_leader_user_id uuid references auth.users(id) on delete set null,
  changed_by uuid references auth.users(id) on delete set null,
  event_type text not null check (event_type in ('assigned', 'changed', 'invalidated', 'system_fallback', 'removed')),
  reason text,
  created_at timestamptz not null default now()
);

create table public.team_core_growth_rewards (
  id uuid primary key default gen_random_uuid(),
  bonus_date date not null,
  source_user_id uuid not null references auth.users(id) on delete cascade,
  leader_user_id uuid not null references auth.users(id) on delete cascade,
  source_core_before numeric(30, 12) not null check (source_core_before >= 0),
  source_core_after numeric(30, 12) not null check (source_core_after >= 0),
  source_core_delta numeric(30, 12) not null check (source_core_delta > 0),
  reward_amount numeric(20, 2) not null check (reward_amount > 0),
  depth integer not null check (depth >= 1),
  batch_id uuid,
  created_at timestamptz not null default now(),
  unique (bonus_date, source_user_id, leader_user_id)
);
```

Для суточного командного бонуса в `public.core_accounts` нужен baseline:

```sql
alter table public.core_accounts
add column if not exists team_bonus_base_balance numeric(30, 12) not null default 0 check (team_bonus_base_balance >= 0),
add column if not exists team_bonus_base_at timestamptz;
```

Если `follows` или `feed_subscriptions` еще нет, нужен социальный слой подписок:

```sql
create table public.feed_subscriptions (
  follower_user_id uuid not null references auth.users(id) on delete cascade,
  followed_user_id uuid not null references auth.users(id) on delete cascade,
  source text not null check (source in ('manual', 'team_leader_required')),
  is_required boolean not null default false,
  created_at timestamptz not null default now(),
  primary key (follower_user_id, followed_user_id, source),
  check (follower_user_id <> followed_user_id)
);
```

На уровне API нельзя разрешать удаление подписки с `source = 'team_leader_required'`, пока есть активная командная связь с этим лидом.

## Проверки уровней и вместимости

Источник уровня: `public.user_profiles.level`, который синхронизируется из `public.core_accounts.level`.

Правило уровня:

```text
member.level < leader.level
```

Вместимость команды нужно вынести в функцию, чтобы потом менять формулу без переписывания клиентского кода.

Текущая принятая модель:

```text
leader_capacity(leader_level) = leader_level * 10
member_capacity_cost(member_level) = member_level
team_capacity_used(leader_id) = sum(member_capacity_cost(member.level))
team_capacity_free = leader_capacity - team_capacity_used
```

Пример:

```text
leader level 3 -> capacity 30
team members: level 1 + level 1 + level 2 + level 5 = used 9
free capacity = 21
```

Преимущество этой модели: capacity измеряет нагрузку команды, а не только количество людей. Лид может вести много новичков или меньше участников высокого уровня.

Пользователи уровня `0` не участвуют в командном распределении и не занимают capacity. Назначение в команду запускается после достижения уровня `1`.

При назначении в команду сервер должен проверять:

- лидер существует;
- у лида есть активный профиль;
- `member.level >= 1`;
- `member.level < leader.level`;
- у лида хватает свободной capacity для стоимости нового участника;
- у участника еще нет активного лида или смена лида разрешена;
- последняя смена лида была больше 24 часов назад;
- если пользователь является самым высокоуровневым пользователем системы, его `leader_type = 'system'`.

## Командный бонус

Базовое правило:

```text
daily_team_delta = current_core_balance - team_bonus_base_balance
leader_reward = round(daily_team_delta * 10%, 2)
```

Пример суточного каскада:

1. Daily rate уже начислен всем пользователям.
2. Участник за сутки вырос с `$100` до `$200`; его delta = `$100`.
3. Его лид получает `$10` в Core.
4. Когда очередь доходит до этого лида, его delta уже включает собственный daily rate и `$10`, полученные снизу.
5. Лид лида получает `10%` от delta этого лида.
6. Каскад продолжается до системного лида или пока следующий бонус после округления до `$0.01` не станет `0`.

Важно для реализации:

- командный бонус начисляется в Core лида;
- бонус не забирается у участника, это дополнительное системное начисление;
- каждое начисление должно иметь ledger/audit запись в `team_core_growth_rewards`;
- бонус считается один раз в 24 часа от разницы между текущим Core и `team_bonus_base_balance`;
- после расчета для пользователя `team_bonus_base_balance` обновляется до текущего Core;
- перед сменой лида, если старый `leader_user_id is not null`, нужно сразу рассчитать и начислить старому лиду бонус за накопленный delta участника с момента последнего `team_bonus_base_balance`;
- после такого финального расчета `team_bonus_base_balance` участника обновляется до текущего Core, чтобы новый лид не получил бонус за рост, который произошел при старом лиде;
- лидерский бонус округляется до `$0.01` при расчете каждого суточного шага;
- для ledger лидерских бонусов достаточно `numeric(20, 2)`;
- каскад командного бонуса останавливается на системном лиде или когда следующий бонус после округления до `$0.01` становится `0`;
- начисление должно быть идемпотентным по `bonus_date + source_user_id + leader_user_id`, чтобы повторный job не удвоил бонус;
- job командного бонуса запускается после завершения daily rate;
- обработка идет от нижних уровней к верхним, например по `user_profiles.level asc`, потому что член команды всегда ниже лида;
- пользователи одного уровня не могут быть лидом друг для друга, поэтому внутри одного уровня порядок не влияет на каскад;
- системный лид завершает каскад и не получает запись в пользовательский Core.

Порядок суточного job:

1. Зафиксировать `bonus_date` и `batch_id`.
2. Начислить всем пользователям daily rate и применить реинвестирование.
3. Выбрать пользователей с активной командной связью и `current_core_balance > team_bonus_base_balance`.
4. Обработать их снизу вверх: от меньшего уровня к большему.
5. Для каждого пользователя посчитать `delta = current_core_balance - team_bonus_base_balance`.
6. Если `round(delta * 0.10, 2) > 0`, начислить эту сумму прямому лиду в Core.
7. Записать `team_core_growth_rewards`.
8. Обновить `team_bonus_base_balance` пользователя до текущего Core после учета полученных снизу бонусов.
9. Повторный запуск за тот же `bonus_date` должен быть безопасен и не создавать дублей.

## Точность Core, ставок и начислений

Решение для MVP:

```sql
core_accounts.balance numeric(30, 12)
core ledger core amounts numeric(30, 12)
core_accounts.team_bonus_base_balance numeric(30, 12)
team_core_growth_rewards.source_core_delta numeric(30, 12)
team_core_growth_rewards.reward_amount numeric(20, 2)
daily_core_rate numeric(12, 10) = 0.0006330000
reinvest_percent numeric(5, 2)
```

Почему Core нужен с высокой точностью:

- ставка `0.0633%` в день хранится как `0.0006330000`, чтобы не терять точность коэффициента;
- Core участвует в ежедневном сложном проценте, поэтому его нельзя обрезать до `$0.01` при каждом начислении;
- пример: `$0.01 * 0.000633 = $0.00000633`; если хранить только центы, такое начисление будет округляться в `0`;
- `numeric(30, 12)` дает 18 знаков до точки и 12 после точки, этого достаточно для текущей шкалы уровней и микродолей ежедневного процента;
- Postgres `numeric` хранится переменной длиной, поэтому запас точности не так дорог, как фиксированный широкий decimal;
- Wallet и лидерский `reward_amount` можно хранить до цента как `numeric(20, 2)`, потому что это денежные posting-операции, а не база сложного процента.

Правила расчета:

- все расчеты выполнять в `numeric`, не в float/double;
- daily income считать через точную ставку `numeric(12, 10)`;
- daily posting в Core записывать в `numeric(30, 12)`, без округления до цента;
- Wallet posting округлять до `$0.01`;
- лидерский бонус считать раз в 24 часа от суточного delta Core и округлять до `$0.01`;
- не использовать float/double для финансовых расчетов;
- Wallet можно оставить менее точным, например `numeric(20, 2)`, если он работает как отображаемый платежный баланс;
- уровень считать от `core_accounts.balance`;
- в UI показывать Core в удобном формате, обычно до `$0.01`, но расчет всегда идет от полного значения.

## Алгоритм распределения

Вход:

- `new_user_id`;
- `referrer_user_id`, если есть;
- pending referral metadata.

Шаги:

1. Если `referrer_user_id` есть, попробовать назначить его лидом.
2. Если реферера нет, сразу создать запись в `team_assignment_queue` или синхронно запустить общий подбор.
3. Если назначение к рефереру не получилось из-за уровня или вместимости, искать подходящего лида во всем дереве команды referrer вниз по структуре.
4. В MVP поиск не идет вверх от referrer, чтобы не назначать новичка к лиду со слишком большой разницей уровней без необходимости.
5. Если подходящего лида нет, создать запись в `team_assignment_queue`.
6. Фоновая задача или server action обрабатывает очередь и ищет лида в общей базе.
7. После назначения создать или обновить `team_memberships`; отдельный `team_id` не нужен, команда определяется через `leader_user_id`.
8. Создать обязательную подписку на лида.
9. Записать событие для аудита и будущей аналитики.

Приоритеты общей очереди для будущей версии:

- каждое пустое место в команде дает `+1` к приоритету;
- выше retention участников команды;
- trust лида влияет на приоритет;
- рост команды увеличивает приоритет;
- получение нового участника из очереди дает `-1` к приоритету, чтобы при прочих равных лиды получали участников по очереди;

## Смена лида

Правило MVP:

- пользователь может запросить смену лида не чаще одного раза в 24 часа;
- лид всегда выше уровнем, чем участник;
- у нового лида должно хватать свободной capacity для стоимости участника;
- реферальная связь `referrer -> referral` не меняется при смене лида;
- меняется только активная командная связь;
- обязательная подписка переносится на нового лида.

Серверный поток:

1. Пользователь выбирает нового лида или нажимает автоматический подбор.
2. Сервер проверяет 24-часовой cooldown (ограничение только на смену по инициативе самого пользователя).
3. Сервер проверяет уровень и вместимость нового лида.
4. Если старый `leader_user_id is not null`, сервер сразу рассчитывает и начисляет старому лиду командный бонус по текущему delta участника.
5. Сервер обновляет `team_bonus_base_balance` участника до текущего Core.
6. Сервер обновляет активную `team_memberships`: ставит нового лида или `leader_user_id = null`.
7. Сервер пишет `team_membership_events`.
8. Сервер снимает обязательную подписку со старого лида, если она больше не нужна.
9. Сервер создает обязательную подписку на нового лида.

## Проверка команды при повышении уровня

При каждом повышении уровня пользователя нужно проверять его текущую командную связь.

Правило:

- если у пользователя нет активной командной связи, ничего не делаем;
- если текущий лид - система, можно запустить автоматический поиск нового лида;
- если текущий лид - пользователь и `leader.level > member.level`, связь остается активной;
- если текущий лид - пользователь и `leader.level <= member.level`, старая связь больше невалидна;
- перед заменой ненулевого лида нужно сразу рассчитать и начислить ему командный бонус за накопленный delta участника;
- после финального расчета бонуса нужно обновить `team_bonus_base_balance` участника до текущего Core;
- невалидную связь нужно заменить через `update team_memberships`, не создавая историю в основной таблице;
- событие смены нужно записать в `team_membership_events`;
- после замены нужно запустить поиск нового лида;
- пока алгоритм поиска не реализован, пользователь получает системного лида: `leader_user_id = null`.

Серверный поток:

1. Daily/core/job или RPC повышения уровня фиксирует новый `member.level`.
2. Сервер загружает активную `team_memberships` пользователя.
3. Если `leader_user_id is null`, запускается попытка подбора лида или связь остается системной.
4. Если `leader_user_id is not null`, сервер сравнивает уровни участника и лида.
5. Если `leader.level <= member.level`, текущая связь становится невалидной.
6. Сервер рассчитывает и начисляет финальный бонус старому лиду, если `leader_user_id is not null`.
7. Сервер обновляет `team_bonus_base_balance` участника до текущего Core.
8. Сервер пишет `team_membership_events`.
9. Сервер запускает `assign_new_leader(member_user_id)`.
10. Если подходящий лид найден, активная связь обновляется на нового лида.
11. Если подходящий лид не найден или поиск еще не реализован, активная связь обновляется на системную: `leader_user_id = null`.
12. Обязательная подписка переносится на нового лида или снимается, если лидер стал системой.

Важно: эту проверку нельзя оставлять только на клиенте. Она должна происходить в Supabase/server-side логике там же, где меняется уровень или сразу после пересчета уровня.

## Функционал команды

Команда должна быть не только структурой начислений, но и рабочим пространством, где лид помогает участникам расти: видеть цели, давать задания, направлять, проверять прогресс и создавать социальное доказательство результатов.

### Подписка на канал лида

- При назначении лида участник автоматически подписывается на канал/ленту лида.
- Эта подписка обязательная, пока действует командная связь.
- Посты лида должны иметь повышенный приоритет в ленте участника.
- В профиле команды нужно показать, что участник получает обновления лида автоматически.
- При смене лида обязательная подписка переносится на нового лида.

Форматы канала лида:

- короткие посты и инсайты;
- цели и публичные отчеты лида;
- инструкции для команды;
- разборы частых ошибок;
- подборки челленджей и задач на день/неделю;
- live/reality updates о собственном пути лида.

### Чаты внутри приложения

Нужны два базовых типа общения:

1. Личный чат `лид -> участник`.
   Используется для целей, обратной связи, приватных вопросов, проверки прогресса и коррекции плана.

2. Групповой чат команды.
   Лид модерирует, задает направление, публикует общие задания, отвечает на вопросы, закрепляет полезные сообщения и поддерживает командный ритм.

Минимальный MVP:

- текстовые сообщения;
- системные события: назначение лида, новая задача, выполненная задача, повышение уровня, публичный отчет;
- read state;
- блокировка/жалоба;
- роль лида как модератора группового чата;
- возможность закрепить сообщение лида сверху.

Позже:

- реакции;
- голосовые/видео-сообщения;
- треды по задачам;
- созвоны/эфиры;
- авто-сводка чата для лида.

### Видимость профилей и обмен контактами

Команда должна помогать людям находить друг друга, но приватность должна быть управляемой.

Базовая модель видимости:

- участник видит профиль своего лида;
- лид видит профили прямых участников;
- участники одной команды могут видеть друг друга в командном списке, если у пользователя включена видимость для команды;
- публичные данные профиля видны согласно общим настройкам публичности;
- приватные контакты не показываются автоматически.

Контакты и соцсети:

- в профиле можно добавить ссылки на Telegram, Instagram, YouTube, TikTok, X/Twitter, Facebook, LinkedIn, email и сайт;
- каждая ссылка имеет настройку видимости: `public`, `team`, `leader_only`, `hidden`;
- в команде можно сделать кнопку "Обменяться контактами";
- обмен контактами между участниками лучше делать через подтверждение обеих сторон;
- контакт с лидом можно сделать проще: участник отправляет запрос, лид принимает или отклоняет;
- для безопасности нужен audit события обмена контактами.

Варианты политики:

- MVP: лид и участник видят публичные контакты друг друга; контакты между участниками команды только после взаимного подтверждения.
- Более открытый режим: участники команды видят team-visible контакты друг друга.
- Более приватный режим: все контакты только по request/accept.

Рекомендация для MVP: показывать командный список и публичные достижения, но обмен приватными контактами делать через подтверждение. Это снижает риск спама и давления внутри команды.

### Цели участника и задания от лида

Участник должен иметь возможность сформулировать свои цели, а лид - превратить их в конкретный план действий.

Поток:

1. Участник описывает цель: чего хочет достичь, почему это важно, срок, желаемый результат.
2. Лид видит цель в рабочей панели команды.
3. Лид предлагает набор задач с дедлайнами.
4. Участник принимает задачи или задает уточняющий вопрос.
5. Система отслеживает автоматические задачи.
6. Участник вручную отмечает субъективные/внешние задачи.
7. Лид проверяет результат, принимает, просит доработку или меняет план.
8. Часть прогресса может публиковаться в командный чат или ленту как социальное доказательство.

Типы задач:

- автоматические: сделать `N` челленджей за день/неделю, увеличить Core на `X`, заработать `X`, пригласить `N` рефералов, опубликовать отчет, обновить желания;
- полуавтоматические: заполнить профиль, добавить публичную цель, сделать пост с результатами, пройти обучение;
- ручные: написать план, записать видео-отчет, провести разговор, изучить материал, выполнить внешнее действие;
- публичные: команда видит постановку и выполнение;
- приватные: видят только участник и лид;
- системные: создаются платформой по правилам роста;
- лидерские: создаются конкретным лидом.

Поля задачи:

```text
title
description
assignee_user_id
leader_user_id
goal_id
type: automatic | manual | mixed
visibility: private | team | public
metric_key
target_value
current_value
due_at
status: proposed | accepted | in_progress | submitted | completed | rejected | expired
verification: automatic | leader_review | self_report
```

### Визуальная модель UI

Экран команды можно сделать как рабочую панель, а не просто список людей.

Основные зоны:

1. Верхняя панель команды.
   Текущий лид, уровень лида, свободная capacity, число участников, статус обязательной подписки.

2. Канал лида.
   Последние посты/инструкции лида, закрепленный пост, кнопка перейти в ленту лида.

3. Чаты.
   Быстрый вход в личный чат с лидом и групповой чат команды. У группового чата виден счетчик непрочитанных и закрепленное задание недели.

4. Участники.
   Список прямых участников, уровень, короткий прогресс, публичная цель, кнопка профиля, кнопка запроса контакта.

5. Мои цели и задания.
   Для участника: список задач от лида, дедлайны, прогресс, кнопка отчета. Для лида: фильтр по участникам, просроченные задачи, задачи на проверке.

6. Командный прогресс.
   Сводка за день/неделю: выполнено челленджей, рост Core, новые уровни, новые посты, новые рефералы, streak команды.

Визуальные варианты:

- Kanban: `Новые`, `В работе`, `На проверке`, `Готово`.
- Timeline: путь участника по дням с задачами, отчетами и достижениями.
- Scoreboard: компактная таблица команды с прогрессом и дедлайнами.
- Goal cards: карточки целей, внутри которых живут задачи и отчеты.

Рекомендация для MVP:

- для участника: вкладки `Лид`, `Чат`, `Задачи`, `Команда`;
- для лида: вкладки `Обзор`, `Участники`, `Проверка`, `Групповой чат`;
- начать с задач `automatic` и `manual`, без сложных тредов и без публичности по умолчанию.

## API и клиентские изменения

Клиент:

- добавить чтение `ref` из URL на старте приложения;
- расширить `guestIdentity` полем `pendingReferral`;
- добавить генерацию и показ собственной реферальной ссылки для зарегистрированного пользователя;
- после завершения регистрации отправлять claim pending referral;
- в Social/Teams показать текущего лида, команду, свободные места и статус распределения;
- добавить вход в канал лида, личный чат с лидом, групповой чат команды и задачи участника;
- для лида добавить рабочую панель участников, задач на проверке и группового чата;
- скрыть действие отписки от обязательной подписки на лида или показывать ее заблокированной.

API:

- `GET /api/referrals/me` - вернуть код и ссылку текущего пользователя;
- `POST /api/referrals/claim` - зафиксировать pending referral после регистрации;
- `GET /api/teams/me` - вернуть лида, участников, capacity, очередь;
- `POST /api/teams/change-leader` - сменить лида с проверкой cooldown;
- `GET /api/teams/conversations` - вернуть личный чат с лидом и групповой чат команды;
- `GET /api/teams/tasks` - вернуть задачи участника или задачи команды для лида;
- `POST /api/teams/tasks` - создать задачу участнику от лида;
- `POST /api/teams/tasks/:id/submit` - отправить задачу на проверку;
- `POST /api/teams/tasks/:id/review` - принять, отклонить или вернуть задачу на доработку;
- `POST /api/team-contacts/request` - запросить обмен контактами;
- фоновый обработчик очереди распределения.

База:

- миграция публичных таблиц referral/team;
- RLS для чтения своих связей, команды и публичных данных лидов;
- writes через server-side API или security definer functions;
- индексы по `leader_user_id`, `leader_type`, `status`, `created_at`;
- транзакции для claim referral и team assignment;
- таблицы для командных чатов, сообщений, задач, проверок задач и запросов обмена контактами.

## Пошаговое внедрение

### Этап 1. Зафиксировать продуктовые правила

- Утвердить терминологию: `лид`, `участник команды`, `реферер`, `реферал`.
- Зафиксировать, что пользователи уровня `0` не распределяются в команды, а назначение начинается с уровня `1`.
- Зафиксировать, что fallback внутри команды пригласившего ищет по дереву вниз, а не только среди прямых участников.
- Утвердить, можно ли пользователю вручную выбрать нового лида или только запросить автоматический подбор.

### Этап 2. Pending referral в guest-first onboarding

- Расширить local guest identity в IndexedDB.
- На входе читать `?ref=`.
- Сохранять pending referral без серверной фиксации.
- Сделать idempotent local claim state.
- Не ломать сценарий пользователя без реферальной ссылки.

### Этап 3. Серверная схема

- Создать таблицы `referral_codes`, `referral_edges`, `team_memberships`, `team_assignment_queue`, `team_membership_events`, `team_core_growth_rewards`.
- Создать таблицу или интеграцию для обязательных feed subscriptions.
- Добавить RLS и server-side functions.
- Сгенерировать типы Supabase.

### Этап 4. Claim после регистрации

- После успешного регистрационного челленджа отправлять pending referral на сервер.
- На сервере валидировать код, self-referral, дубли и состояние пользователя.
- Если referral есть, создавать `referral_edges` ровно один раз.
- Если referral нет, пропускать `referral_edges` и запускать распределение через общую очередь.
- Сразу запускать назначение в команду.
- Возвращать клиенту результат: `assigned_to_referrer`, `assigned_to_team_member`, `queued`, `failed`.

### Этап 5. Команды и распределение

- Реализовать проверку командной связи при повышении уровня: если `leader.level <= member.level`, обновлять активную связь и запускать новый подбор.
- Пока автоматический подбор не готов, при невалидной связи назначать системного лида `leader_user_id = null`.
- Реализовать назначение к пригласившему.
- Реализовать fallback к участнику команды пригласившего.
- Реализовать общую очередь.
- Добавить audit events.

## Принятые решения по вопросам

### Распределение

1. Новый пользователь может прийти без реферальной ссылки.
   Решение: после регистрации такой пользователь попадает в общую очередь распределения.

2. Fallback внутри команды пригласившего.
   Решение: искать по всему дереву команды вниз от пригласившего, а не только среди прямых участников. Поиск не идет вверх, чтобы не создавать слишком большую разницу уровней с новым пользователем.

3. Несколько пользователей с одинаковым самым высоким уровнем.
   Решение: у всех таких пользователей лидом считается система.

4. Пользователь попал не к пригласившему, а к другому лиду из-за нехватки места.
   Решение: отдельное предупреждение не обязательно. В UI достаточно показать текущего лида и команду.

### Pending Referral

1. Срок действия pending referral.
   Решение: срок действия не вводим без отдельной причины. Pending referral хранится до регистрации, перезаписи или очистки гостевых данных.

2. Повторный заход по другой реферальной ссылке до регистрации.
   Решение: новая ссылка может перезаписать pending referral. Для аналитики можно отдельно хранить историю касаний, но claim использует последний код.

### Команды и лиды

1. Вместимость команды.
   Решение: capacity команды считается по сумме уровней прямых участников, а не по количеству мест. Базовая capacity лида: `leader_capacity = leader.level * 10`. Стоимость участника: `member_capacity_cost = member.level`. Лид может принять нового участника, если `team_capacity_used + member_capacity_cost <= leader_capacity`.

2. Смена лида.
   Решение: смена лида по инициативе самого пользователя доступна не чаще одного раза в 24 часа.

3. Админы.
   Решение: админов в MVP нет, поэтому admin override не закладываем.

4. Автоматический перенос при росте уровня.
   Решение: если участник становится того же уровня или выше текущего лида, текущую связь нужно обновить и запустить поиск нового лида. Пока поиск не реализован, участник получает системного лида `leader_user_id = null`.

5. Если у лида лида нет места при автоматическом переносе.
   Решение: система сначала перебирает всех кандидатов уровня `N + 1`, где `N` - текущий уровень пользователя.

6. Выбор кандидата при поиске вниз по дереву пригласившего.
   Решение: искать среди кандидатов уровня `N + 1` по скорингу, который учитывает свободную capacity. Если внутри структуры пригласившего нет свободных кандидатов `N + 1`, искать снаружи.

7. Ручная смена команды.
   Решение: пользователь может попробовать перейти в команду к любому пользователю, который выше него уровнем и проходит проверки capacity/ограничений лида.

8. Запланированная смена команды.
   Решение: пользователь может запланировать, к кому хочет попробовать перейти, когда достигнет уровня текущего лида или если текущий лид его исключит.

9. Ограничения лида.
   Решение: лид может ограничить размер своей команды ниже системного лимита, задать минимальный и максимальный уровень участников, где `max < leader.level`, а позже - добавить другие параметры.

10. Исключение участников.
   Решение: лид может исключать участников из своей команды; исключенный пользователь возвращается в автоматический подбор или использует запланированную смену, если она есть.

11. Командное общение.
   Решение: у команды должны быть личный чат `лид -> участник` и групповой чат команды. Лид модерирует групповой чат, закрепляет сообщения и задает командный ритм.

12. Видимость команды.
   Решение: участник видит профиль лида, лид видит профили прямых участников. Видимость участников друг для друга зависит от настроек приватности; для MVP безопаснее показывать командный список и публичные достижения, а приватные контакты открывать только по подтверждению.

13. Задачи от лида.
   Решение: участник формулирует цели, а лид может назначать конкретные задачи с дедлайнами и проверкой. Часть задач проверяется автоматически по системным метрикам, часть вручную участником и лидом.

### Командный бонус

1. Размер бонуса.
   Решение: лид получает в Core `10%` от суточного прироста Core каждого прямого участника команды.

2. Источник увеличения Core.
   Решение: бонус считается от общего delta Core за сутки, неважно, из каких источников появился прирост.

3. Каскад.
   Решение: командный бонус считается раз в 24 часа после daily rate. Обработка идет снизу вверх, поэтому бонусы, полученные снизу, входят в delta лидов и дают следующий `10%` вверх по цепочке до системного лида.

4. Точность Core, ставок и начислений.
   Решение: Core хранится как `numeric(30, 12)`, потому что он участвует в ежедневном сложном проценте. Daily rate хранится как `numeric(12, 10)` со значением `0.0006330000`. Wallet posting и лидерский `reward_amount` хранятся как `numeric(20, 2)`. Лидерский бонус округляется до `$0.01` при расчете суточного шага.

5. Baseline для командного бонуса.
   Решение: в `core_accounts` хранится `team_bonus_base_balance` - предыдущая сумма Core, с которой уже начисляли лидерский бонус. После суточного расчета baseline обновляется до текущего Core пользователя.

### Pending Referral Analytics

1. История referral touches.
   Решение: в MVP достаточно хранить только последний pending referral. Историю касаний можно добавить позже для аналитики.
