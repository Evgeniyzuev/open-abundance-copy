# Abundance Effect: план разработки приложения с нуля

## 1. Суть продукта

Abundance Effect - mobile-first PWA для людей, которые ищут финансовый рост, новый источник дохода, заказы, монетизацию навыков или переход к более свободному формату работы. Пользователь каждый день движется по персональному плану развития: выполняет задания, растит внутренний капитал, визуализирует желаемое будущее, строит связи с людьми и превращает прогресс в социальный контент.

Внутренний ориентир по механикам - легкость ежедневной рутины: карта развития, уровни, понятный следующий шаг, видимый прогресс и регулярные маленькие победы. В публичной коммуникации не сравниваемся напрямую с Duolingo или соцсетями; эти названия нужны только команде для понимания UX-паттернов. Система должна быть недушной: без искусственных задержек, наказаний, навязчивых анимаций, рекламы и ограничений, которые мешают человеку действовать.

Ключевая идея: заменить бесполезный скроллинг и пассивные мечты на короткие ежедневные действия, которые дают пользователю ощущение роста, социальное подтверждение и измеримый прогресс во внутренней валюте `$`.

### 1.1 Первая аудитория

Основные первые пользователи:
- люди, которые ищут основной или дополнительный источник дохода;
- фрилансеры, которым нужны заказы и стабильный поток клиентов;
- инфлюенсеры и создатели контента, которые ищут монетизацию;
- люди, которые хотят сменить деятельность, работать на себя, удаленно и в удобном режиме;
- пользователи, которым важен финансовый рост и понятный путь к желаемому уровню жизни.

### 1.2 Позиционирование

Abundance Effect - инструмент для роста, в первую очередь финансового, и достижения желаемого через ежедневные действия, задания, связи, капитал и социальное подтверждение.

Важно: не строить коммуникацию на голословных обещаниях. Первый опыт должен работать через интерес, действие и доказательства: пользователь проходит первые задания, видит успешные кейсы других людей, понимает механику роста и хочет повторить похожий путь.

## 2. Основные принципы

1. **Каждый день есть понятный следующий шаг.** Пользователь открывает приложение и сразу видит, что сделать сегодня.
2. **Прогресс измерим.** Основная метрика роста - размер Core/Ядра в `$`, уровень и выполненные задания.
3. **Желание превращается в маршрут.** Пользователь описывает желаемое будущее, а система раскладывает его на задачи, привычки, покупки, связи и проекты.
4. **Социальность встроена в рост.** Пользователь может в один клик показать прогресс, достижение, выполненный челлендж или новый образ будущего.
5. **AI - координатор, не просто чат.** AI помогает выбрать следующий шаг, создать задание, связать людей, объяснить механику и собрать контент.
6. **Позитивная мотивация.** Не ломаем пользователя наказаниями; показываем возможности, близость цели и рост будущего образа.
7. **Минимум трения.** Быстрые действия, короткие сценарии, мобильный интерфейс, no ads, no forced waiting.

## 3. Целевая пользовательская петля

### Ежедневный цикл

1. Пользователь открывает приложение.
2. Видит свой уровень, Core, прогресс до следующего уровня и "задание дня".
3. Выполняет короткое действие: чек-ин, челлендж, публикация, вклад в проект, действие по желанию.
4. Получает награду: Core `$`, Liquid `$`, репутацию, прогресс желания, предмет/артефакт для Vision.
5. Видит обновление карты развития и будущего образа.
6. В один клик публикует сторис/пост о прогрессе.
7. AI предлагает следующий лучший шаг.

### Долгий цикл

1. Пользователь формирует образ желаемого будущего.
2. Система превращает желания в дорожную карту.
3. Через задания, связи, навыки и капитал пользователь растет по уровням.
4. Более высокий уровень открывает более сильные задания, возможности, роли и социальный статус.
5. Пользователь начинает не только выполнять задания, но и создавать ценность для других: наставничество, услуги, проекты, челленджи.

## 4. Главные модули приложения

### 4.1 Home / Today

Главный экран после входа. Его задача - дать пользователю ощущение движения за 5 секунд.

Состав:
- уровень пользователя;
- размер Core в `$`;
- прогресс до следующего уровня;
- ежедневный чек-ин;
- 1-3 рекомендуемых задания на сегодня;
- быстрый доступ к желанию, ради которого пользователь сейчас действует;
- кнопка "Поделиться прогрессом";
- короткая подсказка AI.

MVP-решение: один главный экран с карточкой прогресса, заданием дня и быстрыми действиями.

### 4.2 Development Path / Карта развития

Аналог learning path в Duolingo, но для жизненного, финансового и социального роста.

Состав:
- уровни, привязанные к размеру Core;
- этапы/узлы развития;
- заблокированные и доступные задания;
- треки: стартовый путь, желания, социальный рост, навыки, финансы, проекты;
- визуальный прогресс по каждому треку.

Важно: задания можно видеть заранее, чтобы человек понимал горизонт возможностей, но принимать можно только то, что доступно по уровню/условиям.

### 4.3 Wishes / Система желаний

Один из центральных модулей. Пользователь буквально создает образ желаемого будущего.

Состав:
- список желаний;
- стоимость/ценность желания в `$`;
- срок и приоритет;
- визуальный образ желания;
- связь желания с заданиями;
- прогресс накопления/приближения;
- настройка видимости: приватно, публично, только команда;
- автоматическая публикация публичных желаний в ленту, если пользователь включил этот режим;
- AI-разбор: что нужно сделать, кого найти, чему научиться, сколько Core нужно.

MVP-решение:
- создать желание;
- указать название, описание, сумму, категорию;
- видеть прогресс до желания;
- получить 3 предложенных шага от AI или шаблонной логики.

### 4.4 Vision / Образ будущего

Игровой и эмоциональный слой, который делает цели видимыми.

Состав:
- аватар пользователя;
- галерея будущего;
- визуализация желаний;
- виртуальная экономика аватара;
- принцип "будущее = реальность x1000" для мотивации;
- виртуальное Core аватара = Core пользователя x1000;
- виртуальный уровень аватара считается по тем же порогам, что и реальный уровень;
- виртуальные покупки и достижения аватара оплачиваются доходами от виртуального Core;
- пополнения кошелька пользователя у аватара умножаются на x1000;
- социальный просмотр образов будущего других людей.

MVP-решение:
- базовый профиль будущего;
- выбор визуалов из готовых вариантов;
- загрузка пользовательского контента;
- ссылки и инструкции на внешние инструменты генерации;
- позже подключить image generation внутри приложения.

### 4.5 Challenges / Задания

Задания - основной двигатель ежедневного роста.

Типы заданий:
- системные стартовые задания;
- ежедневные задания;
- задания под конкретное желание;
- социальные задания;
- задания на создание контента;
- задания от пользователей;
- проектные задания;
- задания для роста Core.

Модель наград:
- стартовые onboarding-задания дают небольшую гарантированную награду в Core, например `$1`, при подтвержденном выполнении;
- бонус за качество выполнения может начисляться дополнительно в процентах;
- опциональная часть в Liquid `$`;
- репутация;
- случайные награды позже: боксы, кейсы, предметы, бонусы, артефакты;
- доступ к новым заданиям и ролям.
- пользовательские задания и челленджи выплачивают награду на Wallet исполнителя;
- пользователь, создающий задание, заранее вносит обеспечение на автоматизированный кошелек выплат.

MVP-решение:
- каталог системных заданий;
- принятие задания;
- отправка результата;
- автоматическая проверка onboarding-заданий;
- часть заданий может требовать подтверждения/оценки/ревью/действия от других пользователей;
- гарантированное начисление базовой награды в Core;
- подготовка модели данных под будущие бонусы за качество и случайные награды.

### 4.6 Wallet / Core / Levels

Финансово-игровой слой.

Состав:
- Wallet: ликвидный кошелек пользователя;
- Core: неснижаемое Ядро, основа уровня пользователя;
- Core хранится в `$`;
- Liquid `$`: доступный баланс кошелька для переводов, покупок, будущих счетов и сделок;
- уровень зависит от размера Core;
- пользователь стартует с `0`, шкала уровней продолжается после 40 уровня без верхнего предела;
- уровень открывает задания соответствующего уровня, возможности, видимость и рост команды;
- история операций;
- перевод Liquid `$` в Core;
- автоматическое ежедневное начисление пассивного дохода от Core: `0.0633%` в день;
- с реинвестированием это примерно `26%` годовых, `x2` за 3 года, `x10` за 10 лет;
- ползунок реинвестирования: какой процент ежедневного дохода сразу возвращается в Core;
- Afterburn: выключенная по умолчанию настройка, которая направляет часть Wallet в Core прямо перед расчетом и начислением процентов;
- Afterburn настраивается через лимит или неснижаемый остаток в `$`/`%`;
- переводы Wallet -> Wallet между пользователями уже в MVP;
- внутри системы нет комиссий;
- получатель может отменить/вернуть перевод;
- все переводы записываются в системе, позже могут лечь в приватный блокчейн;
- по переводу можно сформировать чек/подтверждающий документ;
- прогресс до следующего уровня.

Базовая шкала уровней уже зафиксирована в `docs/LEVELS.md`: уровни 1-40 от `$2` до `$1,000,000,000,000`.

MVP-решение:
- Core balance;
- wallet balance;
- таблица уровней;
- расчет текущего уровня;
- перевод Wallet -> Core;
- переводы между пользователями;
- настройка процента реинвестирования;
- настройка Afterburn;
- автоматическое ежедневное начисление пассивного дохода.

### 4.7 Social / Соцсеть роста

Соцсеть должна усиливать прогресс, а не отнимать внимание.

Состав:
- публичный профиль;
- подписки;
- команды: каждый пользователь может набрать команду;
- роль `Лидер команды` и нейтральная роль `Участник команды` для связи внутри команды;
- участники собственной команды должны быть ниже лидера уровнем;
- лидер получает процент от роста Core команды, по умолчанию `10%`;
- папки/категории связей;
- лента прогресса;
- автоматические события: новый уровень, выполненное задание, рост Core, новое желание, достижение;
- автоматические публикации получают базовый приоритет в ленте;
- пользовательские посты;
- пользователь может добавить к автоматической публикации свой текст, фото, видео и другое содержимое;
- one-click posting;
- реакции и комментарии;
- простой Direct;
- обмен внешними контактами по клику, если они заполнены в профиле;
- внешние каналы связи: Telegram, видеозвонки, сайт, соцсети.

MVP-решение:
- профиль пользователя;
- подписка;
- команды;
- лента событий;
- публикация достижения в один клик;
- комментарии;
- простой Direct;
- настройка видимости.

### 4.8 Growth Stories / Контент прогресса

Контент создается из фактов роста, чтобы человеку не приходилось каждый день придумывать пост с нуля.

Состав:
- ежедневные snapshots;
- шаблоны сторис;
- карточки "сегодня я сделал";
- карточки роста Core;
- карточки уровня;
- карточки желания;
- экспорт в изображение;
- шеринг во внешние соцсети.

MVP-решение:
- генерация текстовой/визуальной карточки достижения;
- кнопка "опубликовать в ленту";
- кнопка "скачать/поделиться".

### 4.9 AI Coordinator

AI должен быть операционной системой роста.

Роли:
- Mentor: следующий шаг, мотивация, привычки;
- Investor: Core, уровни, финансовые цели;
- Creator: контент и упаковка прогресса;
- Connector: люди, связи, проекты;
- Coordinator: общий план и приоритизация.

MVP-решение:
- расширенный системный контекст;
- база знаний;
- доступ к данным пользователя через tools/functions;
- рекомендации заданий;
- генерация задач под желания;
- AI может только советовать и давать ссылки на доступные задания, контент и пользователей;
- AI может участвовать в беседе с двумя или несколькими пользователями, чтобы свести их по общим целям и интересам;
- действия AI в MVP: общение, рекомендации, ответы по базе знаний, матчинг;
- AI может использовать публичный контекст без отдельного разрешения, а непубличный контекст только по разрешению пользователя;
- желательно использовать бесплатный API по умолчанию и предусмотреть режим, где пользователь подключает собственный API/лимиты.

### 4.10 Offline-first PWA

Приложение должно работать как web/PWA без публикации в App Store и Play Market. Пользователь открывает приложение по ссылке, а при желании добавляет его на главный экран телефона. Базовые экраны и действия должны оставаться доступными даже при слабом интернете или временной недоступности сервера.

Цель offline-first режима: пользователь продолжает ежедневную рутину, фиксирует мысли и прогресс, а приложение синхронизирует изменения позже.

Что можно делать офлайн:
- открывать уже загруженные экраны;
- читать локально сохраненные задачи, заметки, желания, черновики и часть истории прогресса;
- создавать и редактировать заметки;
- отмечать задачи как выполненные локально;
- писать черновики постов/отчетов;
- готовить submission по заданию;
- сохранять действия в локальную очередь синхронизации.

Что нельзя окончательно подтверждать офлайн:
- начисление Core;
- Wallet-to-Wallet переводы;
- выплаты наград;
- изменение балансов;
- любые действия, где нужен серверный ledger, проверка прав или защита от конфликтов.

Техническая модель:
- PWA manifest для установки на главный экран;
- Service Worker для кэша shell/ассетов/части экранов;
- IndexedDB для локальных данных;
- очередь offline-действий с состояниями `pending_sync`, `synced`, `failed`, `conflict`;
- синхронизация при восстановлении связи;
- конфликт-резолюция через серверные версии записей и понятный UI;
- финансовые и reward-события сохраняются офлайн только как намерение/черновик до серверного подтверждения.

Первыми offline-сущностями должны стать заметки и простые отметки задач, потому что они безопасны, легко проверяются и хорошо демонстрируют ценность PWA.

## 5. Информационная архитектура

Нижняя навигация, mobile-first:

1. **Today** - ежедневный вход и следующий шаг.
2. **Path** - карта развития и задания.
3. **Wishes** - желания и образ будущего.
4. **Social** - лента, связи, профиль.
5. **Wallet** - Core, баланс, уровни.

AI может быть:
- отдельной вкладкой с AI-чатом в MVP;
- кнопками помощи на других вкладках: например, кнопка `?` на задании открывает AI-чат уже с контекстом этого задания.

## 6. MVP: первая рабочая версия

Цель MVP: пользователь может зарегистрироваться, создать желание, пройти стартовые задания, получить Core, увидеть уровень и опубликовать прогресс.

### MVP-сценарий

1. Регистрация.
2. Быстрый онбординг: "чего ты хочешь?".
3. Создание первого желания.
4. Получение стартового маршрута.
5. Выполнение первого задания.
6. Начисление Core.
7. Рост прогресса до следующего уровня.
8. Публикация достижения в ленту.

### MVP-модули

- Auth.
- User profile.
- Core/Wallet balances.
- Levels.
- Wallet-to-Wallet transfers.
- Reinvestment slider.
- Afterburn setting.
- Wishes.
- Teams.
- Starter Challenges.
- Today screen.
- Basic Social Feed.
- Comments.
- Simple Direct.
- Growth post/share.
- Basic AI assistant with knowledge base.
- PWA installability.
- Offline notes screen.
- Offline action queue.
- Automated onboarding verification.
- Peer confirmation/review mechanics for selected tasks.

## 7. Поэтапный план разработки

### Первый практический шаг: Offline Notes Pilot

Цель: собрать самый маленький проверяемый PWA-скелет и доказать, что приложение работает онлайн и офлайн.

Экран: `Notes`.

Функции первого экрана:
- открыть экран заметок онлайн;
- создать заметку;
- отредактировать заметку;
- удалить заметку;
- сохранить заметки локально в браузере;
- открыть экран без интернета после первого посещения;
- создать/изменить заметку офлайн;
- показать статус синхронизации: `local`, `pending_sync`, `synced`, `failed`;
- синхронизировать заметки после восстановления связи.

Технические задачи:
- поднять минимальный Next.js/PWA shell;
- добавить PWA manifest;
- подключить Service Worker;
- добавить IndexedDB-хранилище заметок;
- добавить очередь локальных изменений;
- добавить простой API/server action для синхронизации заметок;
- задеплоить на Vercel;
- проверить на телефоне и десктопе.

Что проверяем вручную:

## 11. Рабочий канбан разработки

Этот документ используется как источник правды и мини-канбан. Codex работает итеративно: одновременно в работе находится один главный пункт, чтобы результат было легко проверить.

### Готово

Пункты попадают сюда только после того, как пользователь написал: `Я проверил`, и результат принят.

- [ ] Документ v1: базовая концепция приложения с нуля.
- [x] **Offline Notes Pilot**: один PWA-экран заметок, локальный кэш, offline-создание заметки, синхронизация после восстановления связи, деплой на Vercel.

### В работе

Codex делает только этот пункт. В конце итерации Codex обязан написать, что именно нужно проверить пользователю.

### Очередь

Когда пользователь пишет `Я проверил`, Codex переносит текущий пункт из `В работе` в `Готово`, берет следующий пункт из `Очередь` в `В работе`, при необходимости задает уточняющие вопросы и начинает реализацию.

- [ ] PWA shell: manifest, service worker, installability.
- [ ] IndexedDB storage layer для локальных данных.
- [ ] Offline action queue.
- [ ] Sync API/server action для заметок.
- [ ] Экран задач с локальной отметкой выполнения.
- [ ] Связь offline-отметки задачи с будущей серверной проверкой.
- [ ] Today screen MVP.
- [ ] Core/Wallet визуализация сложного процента.

### Правила итераций для Codex

1. Codex читает текущий пункт `В работе`.
2. Если данных достаточно, Codex реализует его без дополнительного согласования.
3. Если есть риск неверного продуктового решения, Codex задает короткие уточняющие вопросы.
4. После реализации Codex запускает доступные проверки.
5. Codex не тратит токены на визуальную проверку UI: пользователь сам проверяет внешний вид и UX.
6. Codex пишет пользователю список того, что нужно проверить вручную.
7. Пока пользователь не написал `Я проверил`, пункт остается `В работе`.
8. После `Я проверил` Codex переносит пункт в `Готово`.
9. Codex берет первый пункт из `Очередь`, переносит его в `В работе` и продолжает.
10. Если пользователь просит изменить приоритет, Codex переставляет очередь и фиксирует это в документе.
11. Если в ходе разработки найден новый важный пункт, Codex добавляет его в `Очередь`, а не смешивает с текущей задачей.

### Правило репозиториев

Новая версия приложения разрабатывается в отдельном репозитории. Старый репозиторий используется только как источник контекста и проверенных решений.

Рекомендуемая структура:

```text
f:\git\
  abundance-effect\          старое приложение, reference only
  abundance-effect-pwa\      новое приложение, основной repo
```

Правила:
- Codex может читать старый репозиторий, документацию, схемы, UI-паттерны и отдельные функции;
- Codex не должен вносить изменения в старое приложение без отдельного явного запроса;
- все новые файлы, коммиты, деплой и Vercel-настройки относятся только к новому репозиторию;
- старый код можно переносить только выборочно, с адаптацией под новую архитектуру;
- новый проект должен иметь собственные `README`, `package.json`, `.env.example`, структуру приложения и историю коммитов;
- не создавать новое приложение внутри старого репозитория через `.gitignore`, чтобы не смешивать зависимости, деплой и историю изменений.

### Экономика

Уточнено:
- у пользователя есть Wallet и Core;
- Core хранится в `$`, всегда неснижаемое, определяет уровень пользователя и генерирует пассивный ежедневный доход;
- ежедневный доход начисляется автоматически по ставке `0.0633%` в день;
- с реинвестированием это примерно `26%` годовых, `x2` за 3 года, `x10` за 10 лет;
- пользователь управляет ползунком реинвестирования: какой процент дохода сразу возвращается в Core;
- Afterburn выключен по умолчанию;
- Afterburn можно включить на одно начисление через учебное задание с любой суммой больше `0`;
- Afterburn может иметь лимит или неснижаемый остаток в `$`/`%`;
- первые onboarding-задания дают небольшую гарантированную награду в Core, например `$1`, при выполнении;
- позже к заданиям добавляются бонусы за качество выполнения в процентах и случайные награды: боксы, кейсы, предметы, бонусы;
- Wallet-to-Wallet переводы между пользователями входят в MVP;
- внутри системы нет комиссий;
- лимиты переводов пользователь может настроить в кошельке, но это не обязательно для MVP;
- получатель может отменить/вернуть перевод;
- все переводы записаны в системе, позже возможен приватный блокчейн;
- по переводу можно сформировать чек/подтверждающий документ;
- админов нет: с Core взаимодействует только пользователь через пополнение и получение процентов;
- при угоне аккаунта Core нельзя потратить, можно только получать проценты;
- в будущем биометрически привязанные аккаунты возвращаются владельцу после подтверждения биометрии на сертифицированных устройствах;
- пользовательские таски/челленджи выплачивают награду на Wallet;
- создатель задания вносит обеспечение на автоматизированный кошелек для выплат наград;
- счета, инвойсы, смарт-контракты, автоматические сделки, рынок предметов и расширенная торговля откладываются на следующие этапы.

### Уровни

- шкалу из `docs/LEVELS.md` оставляем;

### Желания и Vision

Уточнено:
- видимость желания настраивается при создании и может быть изменена позже;
- для публичных желаний можно включить автоматическую публикацию в ленту;
- модерация через лайки/дизлайки/жалобы: при заданных коэффициентах желание можно убрать из рекомендаций, цензурить при просмотре профиля или показывать после предупреждения;
- модерация желаний не обязательна для MVP;
- визуализация идет пошагово: сначала выбор из готовых вариантов и загрузка пользовательского контента;
- на старте можно дать ссылки и инструкции на внешние инструменты генерации;
- Vision показывает будущее с коэффициентом, например `x1000`, от сегодняшнего Core;
- пример: пользователь с Core `$1,000` и 10 уровнем имеет сима/аватара с виртуальным Core `$1,000,000` и 20 уровнем;
- аватар может делать виртуальные покупки и достижения на доходы от виртуального Core;
- все пополнения кошелька пользователя у сима умножаются на `x1000`.

### Соцсеть

Уточнено:
- лента показывает все: автоматические события, ручные посты, желания, достижения, дневные результаты;
- автоматические публикации сразу имеют приоритет;
- пользователь может добавить к автоматической публикации контент: текст, фото, видео, описание и другое;
- комментарии нужны в MVP;
- нужен максимально простой Direct;
- обмен внешними контактами по клику, если они заполнены в профиле;
- команды обязательны;
- каждый пользователь набирает команду и получает процент от роста Core команды, по умолчанию `10%`.

### AI

Уточнено:
- в MVP желателен бесплатный API;
- желательно дать пользователю возможность перейти на собственный API и тратить свои лимиты;
- нужно изучить возможность взаимодействия через интерфейс платформ OpenAI/Gemini, если это возможно и легально;
- AI в MVP только советует и дает ссылки на доступные задания, контент, пользователей;
- AI может беседовать с двумя или несколькими пользователями, чтобы свести их по общим целям и интересам;
- low-risk действия: общение с пользователем, рекомендации на основе данных пользователя, желаний, уровня, публикаций, ответы по базе знаний платформы;
- публичный контекст можно использовать без отдельного разрешения;
- непубличный контекст используется по разрешению пользователя;
- база знаний AI: документы о платформе;
- стоит рассмотреть память у провайдеров OpenAI, Gemini, Grok/Groq.

### UX

Уточнено:
- сначала делаем вкладку с AI-чатом;
- на других вкладках добавляем кнопки помощи, например `?` на задании перенаправляет в AI-чат с контекстом задания;
- streaks могут быть нескольких типов;
- типы streaks заранее определяются для заданий;
- пользователь заранее выбирает streaks для своих целей;
- важны Core и уровень: нужно показать силу сложного процента и большие суммы, достижимые при долгом реинвестировании;
- уровень = новые возможности, престиж и подтверждение достижений;
- желания цепляют и возвращают пользователя, потому что он сам их выбрал;
- карта = навигатор: движение, прогресс, новые возможности и интересные места.
